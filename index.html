Meta-Headers

Working with several different tools can be a complete pain if the formats don't match up. To remidy this, we are designing a small human readable header to be included in as many file types as possible describing the file contents in a cross-platform, cross-tool way!

The Basics

While having a standard to define the attributes is a good thing™, the standard must be simple to implement, and to that end, extremely minimal declarations can be used. A simple XML file, encoded in UTF-8 unicode would be required only to declare it's encoding and mimetype to be standards compliant.

<!-- meta { "encoding":"utf-8", "mime":"text/xml" } -->

The snippet above is a perfectly compliant line to include in a basic XML document, and as the headers are intended to be transparent to normal parsers to the files, the entire meta block is wrapped in a comment.

We `hide' the headers this way to ensure backwards compatability with non-compliant parsers

Extensions

Sometimes, it may be desirable to include additional information in a header - as we, the authors, cannot possibly expect to know every possible set of data to be included in the attributes for a file, we incorporate an `extensions' mechanism into the standard, whereby unspecified fields can be included.

In the following code block, we include author information with the standard fields

<!-- meta { "version":1.0, "encoding":"utf-8", "mime":"text/xml", "authors":{ "John Vidler", "Stephen Wattam" } } -->

In this example, the metadata is written across multiple lines - the parser ignores whitespace as much as possible, but some corpus formats only have support for single-line comments - in those cases, we are required to condense the long-form metadata into a single line. Thankfully, XML has paired open/close comment markers, so having this long-form example in a real document would be perfectly valid.

One thing to be careful of when using a non-standardised extension is that the version field must be present. This is to ensure that as the standard expands, extensions that are added to the standard to not interfere with those in the file headers.

Paper: Corpus File Meta-Headers - An Open Standard For Corpus Meta Data

With the increasingly large variety of tools available to the modern corpus linguist, there is a pressing need to be able to efficiently and effectively move data from one tool to another with no (or minimal) loss. Though other standards exist for the representation of corpora at some level (such as TEI, or more general standards such as XML), these often presume a limited set of use-cases, offering a top-down restriction on the information within in order to impose reliable structure. These systems serve to restrict the data that can be stored in such formats, and ultimately this limits their usefulness. Our approach focuses on documenting common features, allowing for variation in document format. Such a method permits innovation whilst maintaining maximal compatibility between tools. Herein we demonstrate a method for representing, parsing, and using such documentation in practice.
